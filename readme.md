# Система аутентификации и авторизации

## Описание проекта

Данный проект представляет собой backend-приложение, реализующее собственную систему аутентификации и авторизации на основе **Django Rest Framework (DRF)**. В процессе реализации проекта была разработана кастомная модель пользователя, собственная система JWT-токенов и гибкая система разграничения прав доступа на основе ролей и ресурсов.

Приложение позволяет:
- Регистрировать пользователей (email, имя, фамилия, пароль).
- Входить и выходить из системы (с использованием JWT-токенов).
- Обновлять профиль и выполнять мягкое удаление аккаунта.
- Управлять ролями пользователей.
- Ограничивать доступ к ресурсам на основе ролей и индивидуальных прав (чтение, создание, обновление, удаление, а также возможность работы со всеми объектами или только с собственными).

Система разработана таким образом, чтобы быть расширяемой и легко адаптироваться под различные бизнес-сущности.

## Схема базы данных

Ниже описаны основные модели, используемые в проекте.

### 1. `UserProfile` (пользователь)
| Поле | Тип | Описание                                 |
|------|-----|------------------------------------------|
| `id` | Integer | Первичный ключ                           |
| `name` | CharField(255) | Имя                                      |
| `surname` | CharField(255) | Фамилия                                  |
| `email` | CharField(255, unique) | Электронная почта  |
| `password` | CharField(255) | Хеш пароля                    |
| `is_active` | BooleanField | Состояние аккаунта                       |

### 2. `Roles` (роли)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer | Первичный ключ |
| `name` | CharField(255) | Название роли |

### 3. `UserRole` (связь пользователей и ролей)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer | Первичный ключ |
| `user` | ForeignKey(UserProfile) | Пользователь |
| `role` | ForeignKey(Roles) | Роль |

### 4. `Resource` (ресурс / сущность бизнес-приложения)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer | Первичный ключ |
| `name` | CharField(255) | Название ресурса  |

### 5. `Access` (правила доступа)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer | Первичный ключ |
| `role` | ForeignKey(Roles) | Роль, к которой применяется правило |
| `resource` | ForeignKey(Resource) | Ресурс, к которому применяется правило |
| `can_read` | Boolean | Разрешено читать свои объекты |
| `can_read_all` | Boolean | Разрешено читать все объекты ресурса |
| `can_create` | Boolean | Разрешено создавать новые объекты |
| `can_update` | Boolean | Разрешено обновлять свои объекты |
| `can_update_all` | Boolean | Разрешено обновлять все объекты |
| `can_delete` | Boolean | Разрешено удалять свои объекты |
| `can_delete_all` | Boolean | Разрешено удалять все объекты |

### 6. `ExpiredToken` (список отозванных токенов)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer | Первичный ключ |
| `jti` | CharField(255, unique) | Уникальный идентификатор токена |
| `expires` | DateTime | Время истечения токена |
| `created_at` | DateTime | Дата добавления в чёрный список |

## Система разграничения прав доступа

### Основные концепции
- **Пользователь** может иметь **одну или несколько ролей**.
- Каждая **роль** обладает набором **правил** для каждого **ресурса**.
- Правила определяют, какие **действия** (CRUD) разрешены и с какими объектами (своими или всеми).

### Правила доступа (таблица `Access`)

Ниже приведена сводная таблица прав для каждой роли на каждый ресурс.  
**Обозначения:**  
- `✓` – разрешено  
- `✗` – запрещено  
- `(свои)` – разрешено только для объектов, где пользователь является владельцем (поле `owner`).  
- `(все)` – разрешено для любых объектов.

#### Роль `admin` (id=3)
| Ресурс     | read | read_all | create | update | update_all | delete | delete_all |
|------------|------|----------|--------|--------|------------|--------|------------|
| users      | ✓    | ✓        | ✓      | ✓      | ✓          | ✓      | ✓          |
| scoreboard | ✓    | ✓        | ✓      | ✓      | ✓          | ✓      | ✓          |
| shop       | ✓    | ✓        | ✓      | ✓      | ✓          | ✓      | ✓          |

> Администратор имеет все права на все ресурсы.

#### Роль `manager` (id=2)
| Ресурс     | read | read_all | create | update | update_all | delete | delete_all |
|------------|------|----------|--------|--------|------------|--------|------------|
| users      | ✓    | ✓        | ✓      | ✓      | ✗          | ✓      | ✗          |
| scoreboard | ✓    | ✓        | ✓      | ✓      | ✓          | ✓      | ✓          |
| shop       | ✓    | ✓        | ✓      | ✓      | ✓          | ✓      | ✓          |

> Менеджер:
> - **users** – может читать всех, создавать, обновлять только свои записи, удалять только свои, но не может обновлять/удалять всех.
> - **scoreboard** и **shop** – имеет полный доступ.

#### Роль `user` (id=1)
| Ресурс     | read | read_all | create | update | update_all | delete | delete_all |
|------------|------|----------|--------|--------|------------|--------|------------|
| users      | ✓    | ✗        | ✓      | ✓      | ✗          | ✓      | ✗          |
| scoreboard | ✓    | ✓        | ✓      | ✗      | ✗          | ✗      | ✗          |
| shop       | ✓    | ✗        | ✓      | ✓      | ✗          | ✓      | ✗          |

> Обычный пользователь:
> - **users** – может читать только свои данные, создавать, обновлять свои, удалять свои.
> - **scoreboard** – может читать всё, создавать, но не может обновлять или удалять.
> - **shop** – может читать только свои записи, создавать, обновлять свои, удалять свои.

## 1. Аутентификация

### 1.1 Вход (Login)
- **Метод:** `POST`
- **Путь:** `/api/log/in`
- **Тело запроса:**
```json
{
  "email": "user@example.com",
  "password": "yourpassword"
}
```

### 1.2 Выход (Logout)
- **Метод:** `POST`
- **Путь:** `/api/log/out`
```json
{
  "detail": "Successfully logged out"
}
```

### 2.1 Список пользователей
- **Метод:** `GET`
- **Путь:** `/api/users`
- **Права:** Требуется право `read_all` на ресурс `users`
```json
{
  "users": [
    {
      "id": 1,
      "name": "Иван",
      "surname": "Иванов",
      "email": "ivan@example.com",
      "is_active": true
    },
    {
      "id": 2,
      "name": "Петр",
      "surname": "Петров",
      "email": "petr@example.com",
      "is_active": false
    }
  ]
}
```
### 2.2 Получение конкретного пользователя
- **Метод:** `GET`
- **Путь:** `/api/users/<int:pk>`
- **Права:**
  - Если запрашивается свой профиль (`pk` == `request.user.id`) – достаточно права `read` на ресурс `users`
  - Если запрашивается чужой профиль – требуется право `read_all` на ресурс `users`
```json
{
  "user": {
    "id": 1,
    "name": "Иван",
    "surname": "Иванов",
    "email": "ivan@example.com",
    "is_active": true
  }
}
```

### 2.3 Регистрация нового пользователя
- **Метод:** `POST`
- **Путь:** `/api/users`
- **Тело запроса:**
```json
{
  "name": "Иван",
  "surname": "Иванов",
  "email": "ivan@example.com",
  "password": "secret",
  "password_confirmation": "secret"
}
```

### 2.4 Обновление данных пользователя
- **Метод:** `PUT`
- **Путь:** `/api/users/<int:pk>`
- **Права:**
  - Если обновляется свой профиль (`pk` == `request.user.id`) – достаточно права `update` на ресурс `users`
  - Если обновляется чужой профиль – требуется право `update_all` на ресурс `users`
- **Заголовки:** `Content-Type: application/json`
- **Тело запроса:**
```json
{
  "name": "Петр",
  "surname": "Петров",
  "email": "petr@example.com",
  "password": "newpassword",
  "password_confirmation": "newpassword"
}
```

### 2.5 Мягкое удаление пользователя
- **Метод:** `DELETE`
- **Путь:** `/api/users/<int:pk>`
- **Права:**
  - Если удаляется свой профиль (`pk` == `request.user.id`) – достаточно права `delete` на ресурс `users`
  - Если удаляется чужой профиль – требуется право `delete_all` на ресурс `users`
```json
{
  "user": {
    "id": 1,
    "name": "Иван",
    "surname": "Иванов",
    "email": "ivan@example.com",
    "is_active": false
  }
}
```

### 3.1 Назначение роли пользователю (только для администратора)
- **Метод:** `POST`
- **Путь:** `/api/users/change`
- **Права:** Текущий пользователь должен иметь роль `admin`
- **Параметры запроса (query string):**
  - `userid` (обязательный) – ID пользователя, которому меняем роль
  - `newrole` (обязательный) – название новой роли (должно существовать в таблице `Roles`)
- **Пример запроса:** `POST /api/users/change?userid=2&newrole=manager`
```json
{
  "detail": "Role 'manager' assigned to user 2"
}
```
